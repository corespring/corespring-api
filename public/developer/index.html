CoreSpring Platform

The CoreSpring Platform provides access to assessment items that can be embedded into any learning system. These items contain questions/activities for students. Sometimes just one question/activity will be contained within an assessment item, but sometimes several may be included in what is called a 'composite' item.

Developers can find assessment items (searching by grade, Common Core Standard etc) using our REST API and can render the items to their users with the CoreSpring Item Player. All items support computer-based administration, and most can also be printed.

Items have an instructor preview view which allows a teacher to view some extra information about the item, view supporting materials like examples of student work or scoring guides, and to preview how an item would look for a student.

Items have also have a student view - which is how the Item appears to the student when they are 'taking' the item.

The CoreSpring Item Player is launched from the 3rd party system (typically in a frame/iFrame) and it handles the user interaction with the item. When the interaction is complete, the outcome of the user's interaction with the item is stored in the CoreSpring Platform (this interaction data is called an Item Session.) This data is then made available to the calling system at a REST URL, or is posted back to the client via a callback.

You can see some examples of items and our Item Player here.

Organizations, Collections, Items

Organizations and Collections are the means by which the Items are organized within the CoreSpring Platform.

Each Organization represents an institution (for example: developer, publisher, school or distict).

Collections are sets of Assessment Items, for example 'ELA' or 'Math' could be collections of items arranged by subject.

An Organization may be granted access to a number of collections, and may create collections owned by that Organization which can then be shared with other Organizations*

CoreSpring maintains and curates a set of content collections of content which will be made available to all organizations. When you register as a developer Organization you will have access to these collections.

*Content creation and publishing is a feature of CoreSpring that will be introduced after our initial 1.0 release

Integration

There are two primary means of integrating CoreSpring content into your system:

  1. Integration using IMS Learning Tools Interopability (LTI) 1.x (and in future LTI 2.0)
  2. Direct integration using the CoreSpring REST API and Item Player

If you are maintaining or implementing a system that supports LTI (Learning Tools Interopability) standards, then this may be the simplest option for you, and may not require writing any custom code on your system.

If you are not implementing LTI, or just want more power and flexibility in how you integrate with CoreSpring. A direct integration using our APIs can be implemented. You can of course combine both options.

LTI Integration

IMS LTI (Learning Tools Interopability) is a standard for interopability between learning systems.

It is supported by many LMS systems like Blackboard, Moodle and CanvasLMS. It allows these systems to launch external tools like CoreSpring within their platforms. Systems like these LMSes are called Tool Consumers (TC) in LTI, and systems like CoreSpring are called Tool Providers (TP). That is, systems that provide tools that can be used within the TC system.

The flow of events that occurs to deliver items is with the CoreSpring LTI tool is:

  1. CoreSpring Tool is configured as an LTI TP on the TC system
  2. Instructor adds an instance of the CoreSpring Tool to a course
  3. Instructor finds and configures the item to be rendered in the course by the CoreSpring Tool
  4. Student 'takes' the item
  5. Instructor can view the outcome and response for each student's Item Session*

* The TC system needs to support LTI outcomes service for the outcomes to be sent back to the TC system

Direct Integration

Direct integration using the CoreSpring REST APIs and Item Player gives you a lot of flexibility in how to integrate the content into your system.

The steps involved in deploying CoreSpring content in your system are:

  1. Finding an Item
  2. Launching the player to render the interaction with an Item, or render the Item profile/preview for an instructor.
  3. Accessing the outcomes of student's interaction with an Item

We have released some sample code on Github that demonstrates these steps in Ruby and Node.js

Finding an Item

You can use our REST/JSON API to find items, and or render the results of searching to your users. See the API documentation below or look at the sample code for information on finding items.

Launching the Item Player

You can launch an item for an instructor to preview by adding your auth token to the following url:

/testplayer/item/{itemId}/run

Once you have an item which you want to deliver to a student, you can configure some settings for the item and launch the Item Player to deliver the interaction UI. Each individual interaction with an Item is called an Item Session, and data about the settings and outcomes of that session are persisted on the CoreSpring Platform.

To launch an Item for a student, you first create an Item Session by making a POST call to the url:

/items/:itemId/sessions

An Item Session will be returned in the response with a unique identifier property 'id', you then launch the item delivery view by putting this Item Session id in the following URL:

/testplayer/session/{sessionId}/run?access_token={your_token}

Configuring Settings for the Item Sesion

In the POST call to create an Item Session you can pass a property called "settings" that determines some settings for runtime behavior of the session.

See the API documentaiton for creating an item session.

Accessing the Outcomes

When the student interaction is complete, you can access the outcome information by retrieving the Item Session with a GET request:

GET /items/{itemId}/sessions/{sessionId}

Please see the sample code on Github for a demonstration of how above steps work.

Registering to Use the API

To use the Corespring API you must first create and register an Organization in the system.

All calls to the API are performed in the context of this Organization.

Upon registering your organization to access the CoreSpring API you will be given a client_id and client_secret.

To test the APIs on this page, the access_token: "demo_token" may be used. This provides read-only access to some test data. The interactive API widget is pre-configured to use this access token but you can replace it with your own token to access real data.

Items

The Items API allows CRUD operations on Assessment Items.

All the API resources described here return an Item or list of Items and some subset of the properties of Item. This complete model for Item is described at the end of this section.

Getting an Item

Returns the Item for the specified :itemId. Does not return every field (see below).


Returns the Item for the specified :itemId. Returns all properties of the Item model.


Getting a List of Items

Returns an array of all Items available to the caller. These Items are contained within the Collections available to their context Organization.

There may be a lot of items, it is recommended to use parameters available for List Queries to filter/limit results. The default limit will be 50 items and the maximum 200. If the result has more than 200 items an error response will be returned.


Returns an array of all Items for the specified :collectionId.


Returns an array of all Items available for the specified :organizationId.

See List Queries for standard parameters for filtering/limiting, and managing array results.

Item list queries will return the summary properties for the Item model. This behavior can be overridden by specifying which fields to return in the model (see List Queries).

Item Sessions

The Item Session API is used for handling a student's interaction with an item and persisting data gathered as part of that interaction.

Getting an Item Session

Returns the Item Session specified by the :sessionId.

Getting a list of Item Sessions

Returns a list of Item Sessions. The standard List Queries options apply.

Create an Item Session

Returns the Item Session for the specified :itemId.

In the POST call to create an Item Session you can pass a property called "settings" that determines some settings for runtime behavior of the session.

Example:

{ "settings": { "maxNoOfAttempts": 1, "highlightUserResponse": true, "highlightCorrectResponse": true, "showFeedback": true, "allowEmptyResponses": false, "submitCompleteMessage": "Ok! Your response was submitted.", "submitIncorrectMessage": "You may revise your work before you submit your final response." } }

Property Description Default
maxNoOfAttempts The maximum number of attempts a user may have at answering the item. 1 (0 is interpreted as no maximum for unlimited attempts.)
highlightUserResponse Upon receiving a response, highlight the users response in the UI. true
highlightCorrectResponse Upon receiving a response, highlight the correct response in the UI. true
showFeedback Show text feedback for the item. true
allowEmptyResponses Accept submission of responses with no value entered by user. false
submitCompleteMessage Message shown to user once the repsonse has been submitted and interaction is complete. "Ok!"
submitIncorrectMessage Message shown to user when they have submitted an incorrect response and may re-submit. "You may revise your work before you submit it."

Item Session Response Model

The Item Session Response contains response information for each question in the item.

The model for each response depends on the type of interaction, for example a single choice question might have one value in the response model indicating the choice made. A multi-choice item would have an array of values, and ordering question would also have an array (in which the sequence was important).

There may be multiple questions defined in an item. Below is an example of a session response for a composite item with questions of different types.

NB boolean properties for 'correct' and 'complete' will be added to the model

Example Response:

[ { "id" : "question1", "value" : "choice1", "outcome" : { "$score": 1 } }, { "id": "question2", "value": "ChoiceA", "outcome": { "$score": 0 } }, { "id": "question23", "value": [["choiceA", "choiceAA"], ["choiceB","choiceBB"]] , "outcome": { "$score": 1.5 } }, { "id": "question4", "value": "text entered by user" } ]

Organizations

Organizations are objects in the system that represent real-world organizations. For example they might be owned/managed by software developers or school districts.

Organizations can have Content Collections and Users associated with them.

Organizations support hierarchy, so they may have children/parents.

Getting a List of Organizations

Returns an array of all Organizations and child Organizations that are available to the caller. Callers will be making this call in the context of their own Organization.

Params The standard List Queries parameters are available for this operation.

Getting an Organization by id

Returns the Organization for the specified :organizationId.

Getting a default organization

Returns the default context organization.

Collections

Collections are sets of Items and may be associated with Organizations.

Getting a Collection

Returns the Collection for the specified :collectionId.

Getting a list of Collections

Returns an array of all Collections available to the caller (Collections attached to any organization the caller has access to).

Returns an array of all Collections associated with the specified :organizationId.

Params The standard List Queries parameters are available for these operations.
